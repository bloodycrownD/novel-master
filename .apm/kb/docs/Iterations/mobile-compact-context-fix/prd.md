# Mobile 压缩上下文确认与反馈修复 PRD

> **平台**：Android + iOS（`apps/mobile`）  
> **性质**：Bug 修复 + 体验对齐；**不改变** Core 事件编排语义、默认 `startDepth`、事件配置数据结构。

## 背景

会话对话页通过 Header「会话操作」抽屉（`SessionActionsDrawer`）提供 **压缩上下文** 入口。该能力在 `mobile-workspace-rename-menu` 迭代中已更名并验收 **T10**（点按后应出现确认框，确认后按事件配置执行压缩）。

用户反馈：点 **压缩上下文** 后 **无任何可见反应**，消息也未被隐藏。排查结论：

1. **确认框未出现**：实现使用 `Alert.alert`，且在关闭底部 `AppModal` 抽屉后 **同步** 调用；在 React Native 上易出现确认框静默失败，用户感知为「点了没反应」。
2. **用户心智偏差**：用户以为会 **直接隐藏消息**；实际产品语义为 **二次确认 → 发出 `session.compaction.requested` → 按「事件配置」执行**（出厂默认 `hide-message`，`startDepth: 6`）。
3. **反馈不足**：即便压缩逻辑执行完毕，当可见消息不足、或事件未配置隐藏动作时，仍可能 Toast「已压缩」但界面无变化，加剧「功能无效」误解。

Desktop 同类入口已使用 **React 确认弹层**（`ConfirmModal` + state），不在关菜单后立刻调原生 `Alert`，行为更稳定。Mobile 应与之对齐。

## 目标（含成功指标）

| 目标 | 成功指标 |
|------|----------|
| 确认可达 | 从会话操作抽屉点 **压缩上下文**，**100%** 出现可操作的二次确认 UI（非依赖易失败的原生 Alert 时序） |
| 压缩可执行 | 用户在确认后，压缩事件 **实际发出**，消息列表 **刷新**；长会话下可见消息按事件配置减少（默认 depth 规则） |
| 反馈可理解 | 用户能区分：**已压缩并隐藏** / **当前无需压缩（无可见变化）** / **执行失败**；不再出现「完全无反应」 |
| 与 Desktop 一致 | 确认文案、标题与 Desktop「压缩上下文」口径一致；交互为「先确认、再执行」 |

## 用户与场景

| 用户 | 场景 |
|------|------|
| 日常写作用户 | 会话过长时，通过菜单手动压缩上下文，减轻模型输入负担 |
| 调试 / 高级用户 | 在「事件配置」中自定义压缩动作后，用手动压缩验证配置是否生效 |
| 误触恢复用户 | 在确认框点「取消」，不触发任何隐藏或事件 |

## 范围

### 包含

1. **确认交互修复**：会话操作抽屉 → **压缩上下文** 使用 **React 确认弹层**（基于现有 `AppModal` 体系），替代关抽屉后的 `Alert.alert`。
2. **执行链路保持**：确认后仍通过 `eventOrchestrator.emit(session.compaction.requested, { trigger: 'manual' })` 执行；成功后刷新消息列表与 token 标签。
3. **结果反馈增强**：
   - 压缩成功且可见消息数量减少 → Toast「已压缩」（或等价文案）
   - 压缩成功但 **未隐藏任何可见消息** → Toast 明确说明「当前上下文无需压缩」类文案（避免误以为坏了）
   - 部分失败 / 异常 → 保留现有错误 Toast 语义
4. **守卫反馈**：`projectId` / `sessionId` 缺失、Agent 运行中等拦截场景，均须有 **可见 Toast**（禁止静默 return）。
5. **回归**：同抽屉其他项（重命名、查看提示词、隐藏/恢复/删除消息、刷新工作树）行为不退化。

### 不包含

- 修改 Core `hide-message` 深度算法、默认 `startDepth: 6` 或事件配置 schema
- 更改「事件配置」「压缩配置」页面逻辑
- Desktop 端改动（仅作交互参照）
- 将手动压缩改为「无确认直接隐藏」
- 会话列表 ⋮ 菜单新增压缩入口（仍仅在会话操作抽屉）

## 核心需求

1. **可靠二次确认**：点「压缩上下文」后，用户 **必须** 看到标题为「压缩上下文」的确认 UI，含说明「将按照事件配置压缩上下文。是否继续？」及「取消 / 压缩」操作。
2. **确认后再执行**：仅当用户确认后发出压缩事件；取消则零副作用。
3. **列表与指标刷新**：执行成功后刷新会话消息列表与 prompt token 展示。
4. **可理解的结果提示**：根据是否实际隐藏了可见消息，给出不同 Toast，使用户知晓功能已执行或当前无需压缩。
5. **异常可见**：Agent 运行中、scope 缺失、事件执行失败等均有明确提示。
6. **与既有验收对齐**：满足 `mobile-workspace-rename-menu` **T10** 的行为意图，并修复其未覆盖的 RN Alert 时序缺陷。

## 验收标准

### 确认交互（T1–T4）

- **T1** **Given** 会话对话页且 Agent 未运行 **When** 打开会话操作抽屉并点 **压缩上下文** **Then** 抽屉关闭，且 **确认弹层可见**，标题含「压缩上下文」。
- **T2** **Given** 确认弹层已打开 **When** 点「取消」 **Then** 弹层关闭，消息列表无变化，无压缩 Toast。
- **T3** **Given** 确认弹层已打开 **When** 点「压缩」 **Then** 弹层关闭，触发压缩流程（见 T5–T7）。
- **T4** **Given** Agent 运行中 **When** 点 **压缩上下文** **Then** 出现 Toast「Agent 运行中无法压缩」（或现网等价文案），**不** 出现确认弹层。

### 执行与反馈（T5–T8）

- **T5** **Given** 会话可见消息足够多（默认配置下 depth ≥ 6 可匹配） **When** 确认压缩 **Then** 较早的可见消息被隐藏，Toast 为成功类（如「已压缩」），列表已刷新。
- **T6** **Given** 会话较短或压缩后无可见消息被隐藏 **When** 确认压缩 **Then** Toast 明确说明 **无需压缩 / 无变化**（非误导性「已压缩」），列表仍刷新。
- **T7** **Given** 事件执行返回失败 **When** 确认压缩 **Then** Toast 提示失败原因（保留现网「压缩部分失败」语义）。
- **T8** **Given** `projectId` 或 `sessionId` 不可用 **When** 触发压缩入口 **Then** 有可见错误提示，**不** 静默无反应。

### 回归（T9–T10）

- **T9** **Given** 会话操作抽屉 **When** 依次使用聊天重命名、查看提示词、隐藏消息、刷新工作树 **Then** 行为与修复前一致。
- **T10** **Given** Android 与 iOS（若可测） **When** 重复 T1–T3 各 3 次 **Then** 确认弹层 **每次** 均稳定出现（无间歇性静默失败）。

## 风险与待确认项

- **短会话判定口径**：以「压缩前后可见消息数是否减少」作为用户可见反馈依据，不改变 Core 执行逻辑。
- **文案**：「当前上下文无需压缩」为建议文案，实现阶段可与 Desktop Toast 对齐微调。
