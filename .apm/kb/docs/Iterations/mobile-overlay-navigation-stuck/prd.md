# Mobile 菜单/弹层导航卡顿与无法返回 Bugfix PRD

> **类型**：缺陷修复（bugfix）  
> **平台**：Android（真机 / 模拟器）  
> **范围**：`apps/mobile` 内 Modal、抽屉、底部菜单、选择器弹层与 Tab/Stack 导航的交互

## 背景

在 Android 上，用户反馈 **各类菜单/弹层** 存在卡顿与导航异常：

1. 呼出菜单（或菜单 **慢半拍** 即将出现）时，若点击 **底部 Tab**、顶栏按钮或进入 **Stack 子页**，界面可能切换，但随后 **系统返回键/侧滑无反应**，像被 **透明遮罩** 挡住。
2. 同类问题在 **多种入口** 均可复现，包括但不限于：对话 Tab 的 **项目抽屉 / 会话操作抽屉**、**agent管理 ⋮ 菜单**、**BottomSheetMenu**、**模型/agent/正则选择器** 等。

根因归类为 **弹层生命周期与页面导航不同步**（弹层未随 Tab 失焦、Stack 跳转或竞态关闭而可靠卸载），导致「页面已变、遮罩仍在」。本 PRD 只描述 **产品行为与验收**；技术方案另文 SPEC。

## 目标（含成功指标）

| 目标 | 成功指标 |
|------|----------|
| 返回可用 | 任意上述场景后，Android **返回键/侧滑** 在 1 秒内恢复预期行为（返回上一级或退出 App），**不得**长期无响应 |
| 无幽灵遮罩 | 切换 Tab 或进入 Stack 子页后，**无**不可见/半透明层拦截点击与返回 |
| 竞态可恢复 | 菜单 **动画尚未结束** 时用户已切走，仍不得进入「永久卡死」状态 |
| 全 App 覆盖 | 所有 Modal/抽屉/底部菜单/选择器弹层纳入同一套 **关闭与导航协调** 策略 |
| 体验可接受 | 菜单可「先关再导航」或「导航时自动关层」——两种均可，以 **不卡死、返回可用** 为准 |

## 用户与场景

| 用户 | 场景 |
|------|------|
| 日常写作用户 | 打开会话 ☰ 后改点「我的」查看设置，再返回对话 |
| 多任务用户 | 项目抽屉慢开时误点底部 Tab，期望仍能正常返回 |
| 配置用户 | agent管理 ⋮ 菜单打开时进入 Agent 编辑器或切 Tab |

## 范围

### 包含范围

1. **Android** 上全 App **弹层类 UI** 与 **Tab / Stack 导航** 的协调修复，至少覆盖：
   - `BottomSheetMenu`（Agent ⋮、项目抽屉内 ⋮ 等）
   - `ProjectDrawer`（会话列表 ☰ 项目抽屉）
   - `SessionActionsDrawer`（对话页 ☰ 会话操作）
   - 选择器：`ModelPickerModal`、`AgentPickerModal`、`RegexGroupPickerModal`
   - `TextPromptModal`（重命名等）
   - `MessageActionMenu`（长按消息菜单）
   - 各 Screen 内 **本地 `visible` 状态** 管理的同类 Modal
2. **Tab 切换**（对话 ↔ 我的）时：相关 Tab 上打开的弹层应 **可靠关闭** 或 **不再拦截** 新页面交互。
3. **Stack 跳转**（如 我的 → agent管理、Agent 编辑器）时：源页面弹层不得 **泄漏** 到新页面。
4. **系统返回**：与现有 `useAndroidChatBackHandler` 等逻辑 **不冲突**；弹层关闭后返回链仍正确。
5. Android 手工回归（见验收标准）。

### 不包含范围

- iOS（本期未指定）。
- 菜单 **打开速度/动画性能** 的单独优化（除非为消除竞态所必需）。
- UI 改版、菜单项增删、业务逻辑变更。
- Core / CLI / Web。
- 技术实现细节（SPEC 另文）。

## 核心需求

1. **导航时卸弹层**：用户从弹层所在页 **切换 Tab** 或 **push Stack 子页** 时，该页关联弹层必须关闭或销毁，**不得** 在目标页继续占用返回栈/触摸拦截。
2. **返回键优先关层**：弹层可见时，系统返回 **应先关闭弹层**；关层后再次返回才走页面级逻辑（与 PRD「对话返回列表」等既有行为兼容）。
3. **竞态安全**：用户在「点击打开菜单 → 菜单尚未完全 visible」期间切 Tab/点按钮，不得进入 **返回无响应** 状态；允许短暂视觉异常，但须 **可自愈** 或通过一次返回恢复。
4. **嵌套弹层**：项目抽屉内 ⋮ + `TextPromptModal` 等 **嵌套** 场景，外层导航时 **一并** 清理，避免只关外层留内层。
5. **统一策略**：同类弹层采用 **一致** 的关闭时机（如 Tab blur / Stack blur），避免仅修单个页面漏网。
6. **最小业务影响**：修复不改变菜单功能项、数据读写与 Agent/会话语义。

## 验收标准

### 通用（每种弹层类型至少抽测 1 次）

- **Given** 用户打开某一弹层/抽屉/菜单（完全可见）  
- **When** 不手动关层，直接点击 **底部另一 Tab**  
- **Then** 目标 Tab 可正常操作；**按返回键/侧滑** 行为符合该 Tab 层级（非永久无响应）；**无** 需多次乱点才能恢复的情况

- **Given** 用户打开某一弹层  
- **When** 通过 **我的 → 配置** 等入口 **进入 Stack 子页**  
- **Then** 子页可交互；返回 Stack 正常；**无** 透明层挡返回

### 竞态（慢菜单）

- **Given** 用户点击打开菜单/抽屉（可感知有延迟）  
- **When** 在菜单 **尚未完全出现** 时点击 **底部 Tab** 或顶栏 **进入其他页**  
- **Then** 最终界面可正常点击；**连续按返回键 3 次内** 必须出现可预期的层级回退或退出，**不得** 全程无响应

### 返回键（主症状负向）

- **Given** 用户曾触发「切 Tab + 菜单」类操作  
- **When** 在 **当前可见界面** 按 **系统返回键/侧滑** 一次  
- **Then** **不得** 出现「完全无反应」；应至少关闭残留弹层、返回上一页或退出 App 之一

### 分场景清单（Android 手工）

| ID | 场景 | 操作 | 通过条件 |
|----|------|------|----------|
| T1 | 对话 · 会话列表 | 打开 **项目抽屉** → 切 **我的** Tab | T1a：我的可点；T1b：返回可用 |
| T2 | 对话 · 会话内 | 打开 **会话操作抽屉** → 切 **我的** | 同 T1 |
| T3 | 对话 · 会话内 | 打开 **切换模型/agent** picker → 切 Tab | picker 不挡新 Tab |
| T4 | 我的 | 打开 **当前模型/agent** picker → 切 **对话** | 同 T1 |
| T5 | agent管理 | ⋮ **BottomSheetMenu** → 进 **Agent 编辑器** 或切 Tab | 编辑器/Tab 可返回 |
| T6 | 对话 | **长按消息菜单** → 切 Tab | 无挡返回 |
| T7 | 项目抽屉 | 抽屉内 ⋮ → **重命名** Modal 打开 → 切 Tab | 嵌套层不泄漏 |
| T8 | 回归 | T1–T7 各做 1 次后，App **杀进程重启** | 无持久化坏状态 |

### 负向（修复前缺陷不得复现）

- **When** 菜单已呼出或正在呼出，用户切 Tab/进子页  
- **Then** **不得** 长期出现「返回键/侧滑完全无反应」
