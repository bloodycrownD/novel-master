# Desktop 主页面 Shell 原型 PRD

## 背景

Novel Master 已在 `apps/mobile` 实现「项目 → 会话 → 聊天」嵌套导航，以及会话工作区 VFS 文件管理能力。Electron desktop（`apps/desktop`）目前仅有空白壳窗口，缺少与产品方向一致的主界面布局。

产品草图定义三栏主界面：

1. **左侧 — 文件预览区**：展示当前选中文件的预览内容  
2. **中间 — 资源管理器**：类似 VS Code 的目录树，与会话工作区联动  
3. **右侧 — Chat 管理区**：类似 mobile 的嵌套结构（项目列表 → 会话列表 → 聊天），并与资源管理器联动  

本期目标是在 Electron 中先交付**可运行的 UI 原型**，验证桌面端信息架构与布局，为后续接入 `@novel-master/core` runtime 做准备。

## 目标（含成功指标）

| 目标 | 说明 |
|------|------|
| 三栏主 shell | 左/中/右区域在桌面窗口内稳定呈现，比例可调整或固定 |
| 布局对齐草图 | 区域职责与 mobile 心智一致，开发者可对照 mobile 理解各栏用途 |
| 可演示 | 双击/命令可启动 Electron，主页面无需额外配置即可看到完整 shell |
| **成功指标** | Win/macOS 上 `npm run desktop:dev` 打开即见三栏；点击各栏控件有可见选中/焦点反馈 |

## 用户与场景

| 角色 | 场景 |
|------|------|
| 产品/开发者 | 在桌面端快速查看主界面布局是否符合预期 |
| 后续迭代开发者 | 以本 shell 为容器，逐步替换 mock 为 core 数据与真实 VFS/聊天 |

## 范围

### 包含范围

- `apps/desktop` 主窗口 UI：**三栏布局**（文件预览 / 资源管理器 / Chat 管理）
- **静态占位内容**：各栏展示示例文案或假列表项，标明区域用途
- **基础选中态**：点击列表项、树节点、Tab 等有视觉反馈（高亮/边框），**无真实数据流**
- 顶栏或侧栏 **最小 chrome**（如应用标题、当前上下文占位），满足「主 shell」可识别
- 参考 `apps/mobile/src` 的信息结构命名与分区（项目/会话/聊天、VFS 树），**仅 UI 层面借鉴**，不移植 RN 组件
- 跨平台：同一套前端资源在 Windows / macOS Electron 下均可展示

### 不包含范围

- 接入 `@novel-master/core` runtime、`novel.db`、SKSP、真实项目/会话 CRUD
- 真实 VFS 目录树、文件读写、左侧预览渲染真实文件内容
- 聊天消息收发、Agent 运行、Composer 输入、流式输出
- 设置、Provider、Agent 编辑等 **二级页面**（无入口或不做）
- 资源管理器与 Chat 区的 **真实联动**（选中会话切换工作区目录等 — 仅静态分区，不做状态同步逻辑）
- 主题系统、国际化、无障碍专项、打包分发（electron-builder 等）
- 自动化 E2E（可选 smoke：窗口能打开即可，不强制）

## 核心需求

1. **三栏主布局**：左（预览）、中（资源树）、右（Chat 嵌套导航）同时可见；默认尺寸下无严重重叠或不可读。
2. **右侧 Chat 区结构**：体现「项目 → 会话 → 聊天」层级占位（可为并列面板、面包屑或分步视图，以可理解为度）。
3. **中间资源管理器占位**：树形或列表形静态节点，风格参考 VS Code 侧栏密度即可。
4. **左侧预览占位**：固定示例内容（如「选中文件后将在此预览」），可选展示假文件名标题。
5. **选中态反馈**：用户点击占位项时，当前项有明确 active 样式；不要求跨栏联动。
6. **Electron 集成**：替换或扩展现有 `renderer/index.html` 壳，通过 `desktop:dev` 可启动查看；不修改 main/preload 业务逻辑（除非加载新静态资源所必需）。

## 验收标准

### 启动与布局

- [ ] **Given** 仓库根目录，**When** 执行 `npm run desktop:dev`（或等效 workspace 脚本），**Then** Electron 窗口打开并显示三栏主界面。
- [ ] **Given** 主界面已打开，**When** 目视检查，**Then** 左、中、右三区均存在且标签/占位文案可辨认（文件预览、资源管理器、Chat 相关）。

### 静态占位与交互

- [ ] **Given** 右侧 Chat 区，**When** 查看占位结构，**Then** 可识别项目列表、会话列表、聊天区域中的至少两层（文案或分区标题即可）。
- [ ] **Given** 中间资源管理器，**When** 查看占位，**Then** 存在树形或列表形假节点（≥3 项）。
- [ ] **Given** 左侧预览区，**When** 未做文件联动，**Then** 显示明确占位说明或示例预览块。
- [ ] **Given** 用户点击任一栏内的占位列表项，**When** 点击完成，**Then** 该项呈现选中/active 样式（同栏内唯一或符合设计即可）。

### 边界（本期不做）

- [ ] **Given** 本期原型，**When** 尝试创建项目、打开真实文件、发送消息，**Then** **无** 上述能力（无报错崩溃即可；按钮可 disabled 或不存在）。
- [ ] **Given** 代码库，**When** 检查 `apps/desktop` 依赖，**Then** **无** 新增对 `@novel-master/core` 业务 runtime 的接线（允许继续仅依赖现有 desktop package 声明）。

### 参考一致性

- [ ] **Given** 对照 `apps/mobile/src/screens/tabs/ChatTabScreen.tsx` 与 VFS 相关组件，**When** 产品/开发者评审，**Then** 右侧与中间栏的**职责划分**与 mobile 一致，不要求视觉像素级一致。
