# 聊天消息编辑多行输入 PRD

> **平台**：Android Mobile（主验收面；与 Android 共用组件时 iOS 顺带一致，不单独立项）  
> **状态**：首轮实现未达验收，需按修订 SPEC **返工** `MessageEditModal`  
> **分支建议**：`fix/mobile-message-edit-multiline`

## 背景

Mobile 聊天中，长按消息 →「编辑」进入「编辑消息」弹窗。历史上该路径使用通用单行 `TextPromptModal`：

- 输入区高度固定，长消息只能单行横向扫视；
- `returnKeyType="done"` + `onSubmitEditing` 使回车等价于保存/收键盘，与底部 **消息输入框**（`ChatComposer`：多行、回车换行、仅点发送提交）体验不一致。

已按迭代 SPEC 新建 `MessageEditModal` 并接入 `ChatConversationPanel`，但 **Android 真机验收未通过**（多行高度/滚动、回车收键盘、键盘遮挡保存按钮等）。根因是技术方案偏离了项目内已验证的 `ChatComposer` 模式，并误用 `ScrollView` 包裹 `TextInput`——属 React Native 常见坑，非产品需求错误。

本 PRD 在 **产品目标不变** 的前提下，明确返工范围与验收标准。

## 目标

| # | 目标 | 可观察成功指标 |
|---|------|----------------|
| G1 | **可读性** | 含 ≥8 行文本的消息，打开编辑后输入区默认可见约 4 行以上；无需单行横向滚动阅读 |
| G2 | **输入习惯一致** | 连续按回车 3 次 → 文本出现 3 个换行，**键盘保持打开**，弹窗不关闭、不自动保存 |
| G3 | **操作可达** | 键盘弹出时，「取消」「保存」在常见竖屏设备上 **可见且可点** |
| G4 | **范围收敛** | 仅聊天「编辑消息」；会话重命名、项目名等 `TextPromptModal` 场景保持单行 |
| G5 | **保存语义不变** | 仍走 `handleSaveMessageEdit` → `updateContent`；隐藏/回滚/Fork 等无回归 |

## 用户与场景

| 用户 | 场景 |
|------|------|
| 日常聊天用户 | 纠正较长用户消息、补充多段说明 |
| 创作/续写用户 | 编辑含大纲、列表、多段落的 prompt，需在弹窗内看清全文再改 |

## 范围

### 包含

- **入口**：聊天 Tab → 长按可编辑消息 →「编辑」→ `MessageEditModal`
- **交互**：
  - 多行纯文本输入；默认可视高度明显高于单行（约 4 行等价，≥120dp）
  - 内容增多时输入区增高，达上限后在 **输入框内部** 纵向滚动（弹窗整体高度受控，不把按钮挤出屏幕）
  - 回车：**插入换行**；不因回车保存、不收键盘
  - 保存：**仅**「保存」按钮；空白/仅空白不可保存（trim 规则与现网一致）
- **实现**：返工既有 `MessageEditModal`；**不修改** `TextPromptModal` 及其它调用方

### 不包含

- `ChatComposer`（底部输入框）行为变更
- 富文本 / Markdown 所见即所得
- 扩展编辑 assistant / tool 消息（当前 `editableTextFromMessage` 不支持则不改）
- 为本需求 **新增** `react-native-keyboard-controller` 等全局键盘库（见 SPEC 分层策略；真机仍不达标时再单独立项）
- Core / CLI / Desktop / Web 消息编辑 UI

## 核心需求

1. **多行默认可视区**：`minHeight` 不小于约 4 行（建议 120dp @ fontSize 16），与 `ChatComposer` 字号/padding 视觉协调。
2. **长文内部滚动**：`maxHeight` 封顶后由 `TextInput` 自身滚动，弹窗 `maxHeight: '85%'` 保证标题与操作栏常驻。
3. **回车 = 换行**：与 `ChatComposer` 一致；禁止 `returnKeyType="done"`、`onSubmitEditing` 保存路径。
4. **键盘不挡操作**：聚焦输入、键盘弹出后，用户无需先关键盘即可点「保存」或「取消」。
5. **打开体验**：`autoFocus`；光标行为保持现状即可。
6. **其它弹窗不受影响**：重命名等仍为单行短输入。

## 验收标准

### 布局与浏览

- [ ] **A1** 打开含单行长句（>40 字）的用户消息编辑，输入区多行展示，无需单行横向滚动即可阅读全文。
- [ ] **A2** 打开含 ≥8 行换行的消息，输入区至少显示约 4 行可见区域，其余在输入框内滚动查看。
- [ ] **A3** 弹窗在常见竖屏高度下，「取消」「保存」始终可见、可点，不被输入区挤出屏幕外。

### 键盘与回车（Android 真机必测）

- [ ] **B1** 编辑框聚焦后，按软键盘 **回车** 一次 → 文本增加一个换行符，**键盘未关闭**。
- [ ] **B2** 连续按回车 3 次 → 3 个换行，无自动保存、无弹窗关闭。
- [ ] **B3** 键盘打开状态下，无需先 dismiss 键盘即可点击「保存」或「取消」。
- [ ] **B4** 仅点击「保存」后消息更新、弹窗关闭；API/DB 与改前一致（含 trim：保留内部换行、去掉首尾空白）。

### 回归

- [ ] **C1** 会话重命名、新建项目等 `TextPromptModal` 仍为单行短输入。
- [ ] **C2** Android 系统返回键：编辑弹窗打开时优先关闭弹窗（`messageEditOpen` 逻辑不变）。
- [ ] **C3** 保存空白或仅空格不可保存（按钮置灰或无效）。

### Given / When / Then

- **Given** 用户消息为多段纯文本  
  **When** 长按 → 编辑 → 在中间按回车插入换行 → 保存  
  **Then** 列表显示新换行，且仅保存一次。

- **Given** 编辑弹窗已打开  
  **When** 用户只按回车、不点保存  
  **Then** 弹窗不关闭、消息列表未变更。

## 与首轮实现的差异（产品层无变更，技术层纠偏）

| 项 | 首轮（未过验收） | 本 PRD 要求 |
|----|------------------|-------------|
| 产品行为 | 多行 + 回车换行 + 按钮保存 | **相同** |
| 输入区布局 | `ScrollView` 包 `TextInput` | **禁止**；对齐 `ChatComposer` 直接 `minHeight`/`maxHeight` |
| 回车 API | 仅 `blurOnSubmit={false}` | 显式 `submitBehavior="newline"`（RN 0.85） |
| 键盘避让 | 居中 Modal + Android 禁用 KAV | 分层策略见 SPEC（底对齐优先，不默认引新库） |
