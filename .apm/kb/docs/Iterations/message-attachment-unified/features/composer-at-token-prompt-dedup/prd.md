# composer-at-token-prompt-dedup Feature PRD

## 背景

父级 `message-attachment-unified` 已经把常驻工作区、规则变更、`@` 引用和用户改稿操作统一进消息附件，并规定：气泡只看用户原文和附件元数据，给模型看的内容在准备提示词时另拼，不写回正文。`composer-ops-chip-lifecycle` 又做成双条 chip——框外无叉状态条（规则变更 + 改稿操作），框内有叉附件条（仅 `@`）。

用下来仍有几处别扭。文件引用同时存在「选择器进 chip」和「正文手输 `@`」两条路，选择器确认后并不往输入框插字。状态条占框外一行，和附件条拆成两套视觉。提示词侧对同一文件路径多次出现时仍会反复附上全文，token 浪费，也和「前文已经给过」的直觉不符。历史上为了展示用户改稿，还曾把操作 XML 写进正文再在列表里抠出来折成工具卡；父级虽改成新路径走附件，仍保留旧会话的正文抠卡兼容，和新的「展示 / 提示词分家」合同打架，也妨碍以后加 skill 一类引用时抄错样板。

本 Feature 在父级之上收口 Composer 交互与提示词降级规则，并去掉旧 UA 正文抠卡兼容。

## 目标（含成功指标）

1. 用户用一种方式管理文件引用：输入框里的彩色 `@路径`（含选择器与手输搜索），删字即取消引用。
2. 状态类信息（规则变更、改稿操作）仍用不可叉的 chip 表达，但不再单独占框外一条 bar；与引用在视觉上分开。
3. 提示词里，文件路径按「可见消息里谁先出现」决定全文或降级；常驻工作区前缀也参与「已出现」；目录引用每次仍拼树：自身不降级，但仍计入已出现路径。
4. 旧会话里靠正文抠 `<action>` 折成的工具卡不再出现；这类消息按普通消息展示。
5. 给人看的界面与给模型（含查看提示词、token 统计）看的内容，继续分家：不把提示词产物写进消息正文，再靠正则从正文还原结构。

**成功指标（可判定）**

- Composer 仅保留一条不可叉的状态 chip 行（规则变更 + 改稿操作）；**移除文件引用 attach chip**，文件引用只认输入框里的 `@路径`。
- `@` 按钮全量选择器与手输 `@` 搜索（最多 5 条）确认后，均在输入框插入 `@路径` 文本。
- 同一可见上下文中，同一文件路径第二次及以后：文本文件引用进提示词时为短提示「该文件前文已引用，无需读取或加载」；规则变更进提示词时不写出该路径内容（路径仍可落在消息附件里；状态 chip 仍可显示）。图片/二进制不套该短提示，仍按 filename 档，但路径仍算「已出现」。
- 常驻工作区前缀已包含的路径，其后消息中的同路径文件引用 / 规则变更按「非首次」处理。
- 目录 `@` 每次拼目录树：自身不降级，但仍计入已出现路径。
- `undo_send` / 编辑回填：**仅正文 `@路径` + 状态投影**（界面上已无文件引用 chip）。
- 历史 UA 两段消息不再合成工具卡；列表按普通用户/助手消息处理。
- 查看提示词、token 统计与 Agent 发送在「首次 / 降级」结果上口径一致。

## 用户与场景

| 角色 | 场景 |
|------|------|
| 作者引用文件 | 点 `@` 打开选择器或多选确认，或手输 `@` 出搜索点选 → 输入框出现彩色 `@路径`；删掉该段文字后引用取消 |
| 作者改规则 / 改稿 | 状态 chip 出现在输入区附件条位置且不可叉；发送后随状态消失；不因删输入框文字而否认真实变更 |
| 作者多轮再 `@` 同一文件 | 第一轮提示词含全文；后续轮次同路径多为短提示；置位后可见范围变了，重新按可见消息数「第一次」 |
| 作者撤销刚发的纯用户消息 / 编辑回填 | 输入框恢复当时正文（含 `@路径`）与状态 chip；界面上无文件引用 chip |
| 作者打开旧会话 | 以前的 UA 工具卡不再出现；对应消息当普通气泡看（可能含历史留下的操作原文） |

## 范围

### 包含范围

1. **Composer 布局**：去掉框外无叉状态 bar；**移除文件引用 attach chip**；框内仅保留规则变更 + 改稿操作的不可叉状态 chip；文件引用只在输入框用 `@路径`。
2. **引用录入**：全量文件选择器与手输 `@` 模糊搜索（最多 5 条）并存；两者都在输入框生成 `@路径`；删除 token 即取消对应引用。
3. **提示词降级（仅提示词消费者）**：按当前可见（未隐藏）消息顺序，对文件路径计数「第几次出现」；常驻工作区前缀先计入已出现。首次附全文；其后文本文件引用用短提示，规则变更不在提示词中写出该路径内容。同条消息同路径时文件引用优先，规则变更不进提示词（判定按附件来源优先级，不依赖落库数组顺序）。改稿操作附件自带内容，不参与这套路径首次判定。目录每次拼树：自身不降级，但仍计入已出现路径。路径比较前做规范化（去目录尾 `/`、补前导 `/`、对齐当前 VFS 逻辑 path 约定）。
4. **短提示文案**（文本文件引用非首次）：带 path 的最小文件块，正文为「该文件前文已引用，无需读取或加载」（不采用带行号的常规文件块形态）。
5. **消费者分家与回填**：气泡 / `undo_send` / 编辑回填只读原文与状态投影（**回填仅正文 `@路径`**）；Agent 发送、查看提示词、token 统计走同一套准备提示词流程；禁止把提示词用的标签正文写进 sqlite 消息正文后再从正文抠结构做 UI。
6. **移除旧 UA 展示兼容**：不再把历史 `user_vfs_action` / ack 折成工具卡；按普通消息渲染。
7. Desktop 与 Mobile 同口径。

### 不包含范围

- 常驻工作区规则引擎本身、Explorer / `$filetree` 实时轨。
- Skill 引用的产品化（可预留附件来源，本 Feature 不交付 skill 选择器）。
- 把历史 UA 消息一次性迁移成新附件结构（接受当普通消息；不做数据升级）。
- 为 Mobile / Desktop 输入框彩色 token 强制同一底层编辑器实现（体验一致即可；允许实现手段不同）。

## 核心需求（相对父级的变更）

1. **状态条挪位与引用入口统一**  
   废止「框外状态条 + 框内可叉 `@` chip」为主交互。状态（规则变更、改稿操作）→ 单行不可叉 chip。文件引用 → 输入框 `@token`。父级「选择器确认后至少进附件草稿 chip」改为「确认后插入输入框 `@路径`」，发送时仍从正文与结构化附件合并出 `attach`（具体合并以发送时扫描与去重为准，不在本 PRD 展开实现）。

2. **手输 `@` 搜索**  
   用户在输入框输入 `@` 后弹出模糊匹配文件列表，最多展示 5 条；点选后插入 `@路径`。与全量选择器并存，结果形态一致。

3. **可见序「首次全文、其后降级」**  
   只根据当前可见消息（外加常驻前缀在最终提示词中的先后）判断路径是否首次出现，不必单独监听置位/压缩事件。文本文件引用非首次 → 短提示；规则变更非首次 → 提示词不显示该路径内容（附件里仍可保留路径；chip 仍可有）。图片/二进制非首次仍按 filename 档，不换中文短提示，但路径仍计入「已出现」。目录自身不降级，但仍计入已出现路径。路径写法差异（有无前导 `/`、目录尾 `/`）在比较前规范化为同一 key。

4. **同条冲突**  
   同一条用户消息上，同一路径既有规则变更又有文件引用时：提示词只按文件引用加载（首次全文或短提示）；规则变更侧不进提示词。处理顺序固定为文件引用优先于规则变更，不依赖附件数组落库顺序。

5. **移除 attach chip；回填仅正文 `@路径`**  
   产品上不再提供文件引用 chip。`undo_send` 与编辑回填只恢复正文（含 `@路径`）和状态类投影。再次发送时仍可从正文 `@` 扫描出引用。

6. **去掉旧 UA 抠卡**  
   废止父级「历史 UA 只读兼容折成工具卡」的验收；旧消息当普通消息。新路径继续禁止再写出 UA 两段消息。

7. **两类消费者合同**  
   明确：界面消费者不得依赖「从消息正文正则提取标签」来还原附件/操作结构；提示词消费者不得把组装结果写回消息正文列。为后续 skill 等引用预留同一纪律。

## 验收标准

### Composer 与引用

- [ ] **Given** 同时存在规则变更与改稿操作，**When** 打开 Composer，**Then** 仅见一条不可叉状态 chip 行（无框外独立状态 bar），且**不存在**文件引用 attach chip。
- [ ] **Given** 用户通过全量选择器选中文件并确认，**When** 回到输入区，**Then** 输入框中出现对应 `@路径` 文本（彩色标识）。
- [ ] **Given** 用户手输 `@` 并看到搜索列表，**When** 列表至多 5 条且点选一项，**Then** 输入框插入该 `@路径`。
- [ ] **Given** 输入框中已有某 `@路径`，**When** 用户删除该段文本，**Then** 发送时不再携带该路径的文件引用。
- [ ] **Given** 用户对一条含 `@路径` 的纯用户消息执行 `undo_send` 或编辑回填，**When** Composer 恢复，**Then** 正文含原 `@路径`、状态 chip 可按投影出现，且界面上**无**文件引用 chip。

### 提示词降级

- [ ] **Given** 可见历史中用户消息首次 `@` 某文本文件，**When** 查看真实提示词或发送，**Then** 该路径以完整文件内容出现在附件段。
- [ ] **Given** 其后又一条可见用户消息再次 `@` 同一文本文件，**When** 查看真实提示词或发送，**Then** 该次为短提示「该文件前文已引用，无需读取或加载」（最小文件块形态），而非全文。
- [ ] **Given** 常驻工作区前缀已包含路径 A，**When** 本轮消息再 `@` A（或写法等价、规范化后相同）或带规则变更 A，**Then** 按非首次处理（`@` → 短提示；规则变更 → 提示词不写出 A 的内容）。
- [ ] **Given** 同一条消息对路径 A 同时有规则变更与 `@`，且落库时规则变更排在 `@` 前面，**When** 准备提示词，**Then** 仍仅按 `@` 规则出现（全文或短提示），规则变更不进提示词。
- [ ] **Given** 用户 `@` 某目录，**When** 多次出现在可见提示词中，**Then** 每次仍拼目录树、不降级为短提示；该目录 path 仍计入已出现，其后同 path 的文件引用 / 规则变更按非首次处理。
- [ ] **Given** 用户 `@` 图片或二进制且该路径非首次，**When** 准备提示词，**Then** 仍为 filename 档形态，不含中文短提示文案。
- [ ] **Given** 查看提示词与发起 Agent 发送，**When** 同一会话同一可见消息集，**Then** 首次/降级结果一致。

### 旧 UA 与分家

- [ ] **Given** 库中仍有历史 UA 两段消息，**When** 打开会话列表，**Then** 不再出现合成的用户改稿工具卡；消息按普通气泡展示。
- [ ] **Given** 任意新发送路径，**When** 检查消息正文列，**Then** 不含为提示词准备的 attachment 套层标签；气泡不依赖从正文抠标签还原附件结构。

## 约束与依赖

- 依赖父级 PRD 与 `composer-ops-chip-lifecycle`、`file-ref-picker-ux`；本 Feature **局部 supersede**：状态条布局、文件引用以 attach chip 为主入口、每次文件引用均全文进提示词、历史 UA 只读折卡（T-UO2 类验收）、以及回填时恢复 attachments chips（T-TX2 旧口径）。
- 保留：attachments 与原文分列、新路径不写 UA 两段、改稿操作自带内容、发送时可空正文但有附件/状态、双端同口径。

## 风险与待确认项

- 历史 UA 改为普通消息后，旧会话可能直接露出操作原文，可读性下降；本 Feature **不接受** 再做抠卡，也不做迁移。
- 文件在非首次之后又被局部编辑：是否仍短提示，本 Feature 默认仍短提示（变更由改稿操作附件表达）；若产品要「编辑后强制再附全文」，另开变更。
- 输入框彩色 token 在 Mobile 与 Desktop 的实现难度不同，以体验一致为验收，允许实现差异。
