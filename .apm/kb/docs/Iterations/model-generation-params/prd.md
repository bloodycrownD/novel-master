# 模型生成参数与思考能力 PRD

## 背景

Novel Master 在服务商配置中支持为每个**已保存模型**设置上下文上限、采样参数（temperature、max_tokens 等）与 Token 计数方式。Agent 运行时通过内部 LLM 协议适配 OpenAI、Anthropic、Gemini 三类服务商发起请求。

当前缺口：

1. **思考（reasoning）能力**：部分模型支持扩展思考/推理输出；产品侧无法在模型配置中显式开启，只能依赖服务商默认行为。响应侧已能展示思考内容，但**请求侧无统一开关**。
2. **生成参数模型分散**：上下文上限服务于内部预算（compaction、计数），采样参数服务于 HTTP 请求，二者同存于模型配置却语义不同；随 thinking 等能力增加，若继续「每能力复制一套 enabled + 协议字段」将导致配置与维护成本上升。

本迭代在**模型配置**域建立统一的**生成参数**产品能力（首版落地 **思考开关**），并理顺与「内部预算类」配置的边界。默认行为与现网完全一致，保证存量用户无感。

## 目标（含成功指标）

| 目标 | 成功指标 |
|------|----------|
| 思考可配置 | 每个已保存模型可单独设置「思考」开/关；默认 **关**（与现网相同） |
| 开启后生效 | 思考 **开** 且用户选用该模型发起 Agent/模型请求时，对应服务商启用思考能力；流式与历史消息中可呈现思考内容（与现网展示能力衔接） |
| 生成参数可扩展 | 产品文档与配置模型明确区分「内部预算」与「生成参数」；thinking 作为生成参数下第一项能力，后续可增参而不重复造轮子 |
| 双端可配 | Desktop 与 Mobile 模型设置页均可配置思考开关；保存后持久化并在下次请求生效 |
| 兼容性 | 未配置或关闭思考的存量模型：**零行为变化**；不要求用户迁移或重新保存模型 |

## 用户与场景

| 用户 | 场景 |
|------|------|
| 配置者 | 为 Claude / Gemini 等支持思考的已保存模型开启思考，写作 Agent 在复杂章节规划时获得更可读的推理过程 |
| 日常用户 | 不修改配置时，一切与现网相同；关闭思考即回到当前行为 |
| 调试者 | 同一模型对比开/关思考时的输出差异与延迟，通过模型设置页切换 |
| 维护者 | 新增服务商能力时，在「生成参数」体系下扩展，而非散落多个独立开关字段 |

## 范围

### 包含范围

1. **模型配置 — 思考开关**
   - 在服务商 → 已保存模型的设置界面新增 **「思考」** 开关。
   - 默认：**关**（字段缺失视为关，与现网一致）。
   - 关：请求行为与现网 **完全相同**。
   - 开：对该模型发起的 LLM 请求启用该服务商支持的思考/reasoning 能力。

2. **生成参数产品分层（首版）**
   - 明确两类配置：
     - **内部预算**：上下文上限、Token 计数方式 — 不直接对应用户可见的「模型生成风格」，继续服务 compaction / 计数。
     - **生成参数**：采样、思考等 — 影响发往服务商的生成行为。
   - 思考归入 **生成参数**；采样保持现有用户可见能力与字段含义 **不变**（temperature、max_tokens 等仍可在采样区配置）。

3. **三协议支持**
   - 对已保存模型所属服务商的 protocol（`openai` / `anthropic` / `gemini`）分别启用思考；用户仅操作统一「思考」开关，不需选择协议。
   - 若某协议/模型组合不支持思考，开启后应有可理解的失败或降级提示（实现阶段定具体文案，验收要求：**不**  silent 失败或挂起）。

4. **配置入口**
   - Desktop：模型采样/设置页增加思考开关。
   - Mobile：模型设置页增加思考开关。
   - 配置持久化于该已保存模型，随模型 id 保存；切换工作区模型后使用各自模型的思考设置。

5. **与 Agent 的关系**
   - Agent 可选用模型 pin 或工作区当前模型；思考设置 **以实际请求的已保存模型** 为准，不按 Agent 单独配置（本迭代不做 Agent 级思考覆盖）。

### 不包含范围

- 思考高级参数 UI（如 budget tokens、reasoning effort 档位）— 首版仅 **开/关**；高级项可 follow-up
- 修改响应侧 thinking 块展示样式或 transcript 存储格式
- 按 Agent、按项目、按会话覆盖思考设置
- 为非已保存模型（仅建议列表）配置思考
- OpenAI 兼容代理的厂商差异穷举文档（实现阶段覆盖主流内置服务商）
- CLI `nm provider model` 子命令完整 parity（可 follow-up）
- 批量为历史已保存模型默认开启思考

## 核心需求

1. **默认关、兼容现网**：思考开关默认关闭；关闭时请求与输出与现网无差异。
2. **按模型持久化**：每个 `providerId/vendorModelId` 已保存模型独立保存思考开/关。
3. **统一产品语义**：用户面对一个「思考」开关；系统按服务商协议映射到具体 API 能力。
4. **生成参数分层清晰**：内部预算（上下文、计数）与生成参数（采样、思考）在产品文案与设置页分区可辨。
5. **双端设置一致**：Desktop / Mobile 同一模型看到相同思考开/关状态。
6. **运行时使用已保存模型配置**：Agent 与直接模型请求均读取该模型持久化配置，不在运行时隐式覆盖。

## 验收标准

### 思考开关 — 默认与兼容

- **Given** 存量已保存模型，用户从未打开思考设置  
  **When** 用户发起 Agent 对话  
  **Then** 行为与现网 **一致**（无新增思考请求、无回归）。

- **Given** 用户在模型设置将「思考」设为 **关** 并保存  
  **When** 使用该模型请求  
  **Then** 与现网关闭状态一致。

### 思考开关 — 开启

- **Given** 用户为某 Anthropic 已保存模型开启「思考」并保存  
  **When** 工作区选用该模型并完成一次 Agent 往返  
  **Then** 可观测到思考类输出（流式或最终消息中的思考内容，与现网展示能力一致）；请求侧已启用思考（相对关闭时可区分）。

- **Given** 用户为某 Gemini 已保存模型开启「思考」并保存  
  **When** 完成一次 Agent 往返  
  **Then** 同上，思考输出可呈现。

- **Given** 用户关闭思考后再次请求  
  **When** 使用同一模型  
  **Then** 回到无思考请求行为。

### 配置持久化与双端

- **Given** 用户在 Desktop 为 `openai/gpt-4o` 开启思考  
  **When** 用户在 Mobile 打开同一已保存模型设置  
  **Then** 思考开关显示为 **开**。

- **Given** 用户修改思考开关并保存  
  **When** 重启应用后再次打开该模型设置  
  **Then** 状态与保存时一致。

### 生成参数分区

- **Given** 用户打开已保存模型设置页  
  **When** 查看页面结构  
  **Then** 可区分「上下文上限 / Token 计数」与「采样 / 思考」等区域；思考开关位于生成参数相关区域，**不** 与上下文上限混为同一开关。

### 不支持组合

- **Given** 某已保存模型所在服务商/模型组合不支持思考  
  **When** 用户开启思考并请求  
  **Then** 用户收到 **可理解** 的错误或提示（非无限加载、非无输出静默失败）— 具体文案实现阶段确定，验收以「可判定失败原因」为准。

## 约束与依赖

- 依赖 `provider-model` 已保存模型、三协议 adapter 与 `ModelRequestService` 调用链。
- 依赖 `agent-system` Agent 运行时选用已保存模型发起请求。
- 响应侧 thinking 内容块与流式事件已存在；本迭代聚焦 **配置与请求启用**。
- 与 `desktop-chat-workspace-polish`、`project-agent-config` **无** 功能依赖，可并行迭代。

## 非功能需求（业务/体验）

- 用户无需理解 protocol 差异即可使用单一「思考」开关。
- 关闭思考时，不增加可感知延迟或额外 token 消耗。
- 设置页保存成功后有明确反馈（与现网模型设置保存体验一致，如「已保存模型设置」类 toast）。

## 风险与待确认项

| 项 | 说明 |
|----|------|
| 协议差异 | OpenAI 兼容层各代理对 reasoning 字段支持不一；首版覆盖内置 seed 服务商，自定义 OpenAI 协议代理行为以实现为准 |
| max_tokens 与思考额度 | 部分服务商思考消耗输出 token 上限；首版仅开关，高级调参 follow-up |
| 多轮对话 | 开启思考后多轮会话是否稳定——依赖现有 thinking 块往返；若现网已支持响应侧，本迭代不扩大范围 |
| 命名 | 产品文案统一为「思考」；英文文档可用 reasoning/thinking 对应 |
