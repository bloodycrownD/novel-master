# 回滚失败降级兜底 PRD

## 背景

消息回滚在 v2 模型下是 **复合操作**：截断锚点之后的消息，并将工作区恢复到锚点 checkpoint 树（`message-checkpoint-v2`）。`vfs-user-ops-unified-tool-turn` 进一步约定：锚点无 checkpoint 时不抛错，用前序 checkpoint 或空树 baseline，**始终**截断 tail 消息。

当前实现中，若 VFS 正向 restore 在事务内失败（如 revision 缺失、路径恢复错误），整次回滚 **事务回滚**，消息与工作区 **均不变**。用户已确认对话、但文件仍停留在「回滚前」状态，缺少明确的补救路径。

与此同时，本迭代 **不** 将「仅删消息」与长按菜单的「删除消息」混为一谈：用户删除单条消息不 reconcile VFS；本需求的降级兜底是 **回滚失败后的显式用户选择**，语义上仍属于回滚流程的一部分。

### 与现有「无 checkpoint 不阻断」的关系

| 场景 | 现有行为 | 本 PRD |
|------|----------|--------|
| 锚点无 checkpoint，前序树可解析 | 不报错，按前序/空树 reconcile + 截断消息 | **不变**（非失败） |
| VFS restore 可判定失败 | 整单失败，无变更 | **新增**降级弹窗 |
| Agent 运行中 / 消息不存在 | 直接拒绝 | **不变**（不进入降级） |

## 目标（含成功指标）

| 目标 | 成功指标 |
|------|----------|
| 失败可感知 | VFS 还原失败时，用户看到 **简要、可读** 的错误说明（非仅 Toast 一闪而过） |
| 用户可控 | 用户可在 **继续（仅删对话）** 与 **取消** 之间选择；取消后状态与发起回滚前一致 |
| 降级语义清晰 | 选择继续后：锚点及以前消息保留，锚点之后消息删除，**工作区文件保持失败前状态** |
| 不误伤 | 未确认前 **不** 部分提交（不删消息、不改文件） |

## 用户与场景

| 角色 | 场景 |
|------|------|
| Mobile / Desktop 对话用户 | 长按消息 →「回滚」→ 确认；因工作区数据问题 restore 失败 → 弹窗说明原因 → 选择仅删后续对话以继续写作，或取消保留完整现场 |
| 写作者 | 对话走错分支但稿文件仍可用；愿牺牲后续聊天记录、保留当前 VFS 内容 |

## 范围

### 包含范围

1. **失败检测**：`rollbackToMessage` 在 VFS reconcile 阶段发生可恢复失败时，向 UI 返回 **可区分** 的失败信息（含错误类型/简要原因），而非仅泛化「回滚失败」。
2. **二次确认弹窗（Mobile + Desktop）**
   - 展示：简要错误说明 + 后果说明
   - **继续**：执行 **仅消息截断**（`deleteAfterSeq` + 清理 tail checkpoint），**跳过** VFS reconcile
   - **取消**：不修改消息与工作区
3. **文案**：避免将「删后续对话」表述为「删文件」；工作区文件在继续路径下 **保持不变**。

### 不包含范围

- 技术方案、API 形状、错误码枚举、事务拆分（见后续 SPEC）
- CLI 交互式降级（可后续单开；本 PRD 以 Mobile + Desktop 为准）
- 修改「无 checkpoint 不阻断」的前序/空树策略
- 恢复长按菜单「删除消息」能力
- VFS 侧「激进删除 tail 改动文件」作为兜底选项
- 自动静默降级（**必须**用户确认）

## 核心需求

1. **失败才弹窗**  
   仅当完整回滚（消息 + VFS）无法完成且错误属于「可降级」类时展示弹窗；门禁类错误（Agent 运行中、消息不存在等）仍直接失败，无降级选项。

2. **继续 = 仅截断对话**  
   用户选择继续后：删除 `seq > anchor` 的消息及对应 tail checkpoint；**不**执行 VFS path reconcile；live 工作区与失败前一致。

3. **取消 = 全或无**  
   用户选择取消或关闭弹窗：消息列表与工作区均与发起回滚前一致。

4. **一次确认链**  
   保留现有回滚 destructive 一次确认；VFS 失败后的降级选择为 **第二次** 确认，文案明确两种后果差异。

5. **成功反馈区分**  
   完整回滚成功 vs 降级仅截断成功，UI 反馈文案可区分（如后者提示「对话已截断，工作区未恢复」）。

## 验收标准

### A. VFS 失败 → 弹窗 → 取消

- **Given** 回滚因 VFS restore 失败（如 revision 缺失）  
- **When** 用户在降级弹窗选择「取消」  
- **Then** 消息列表与工作区文件均与回滚前一致；无 tail 消息被删

### B. VFS 失败 → 弹窗 → 继续（仅对话）

- **Given** 锚点之后有多条消息，且完整回滚因 VFS 失败  
- **When** 用户选择「继续」/「仅删除后续对话」  
- **Then** 锚点及以前消息保留；`seq > anchor` 的消息消失；工作区文件内容与失败前 **逐路径一致**

### C. 完整回滚成功（回归）

- **Given** checkpoint 与 revision 完整  
- **When** 用户回滚并确认  
- **Then** 不出现降级弹窗；消息截断且工作区恢复到锚点树（与 v2 现有验收一致）

### D. 无 checkpoint 不阻断（回归）

- **Given** 锚点为纯文本消息、前序 checkpoint 可解析  
- **When** 回滚  
- **Then** 不视为「失败」，不展示本 PRD 降级弹窗；行为与 `vfs-user-ops-unified-tool-turn` 一致

### E. Agent 运行中（回归）

- **Given** Agent 流式或运行中  
- **When** 用户发起回滚  
- **Then** 直接拒绝；**不**出现降级弹窗

### F. 文案

- **Given** 降级弹窗展示  
- **When** 用户阅读选项  
- **Then** 「继续」选项 **不** 暗示会删除或修改工作区文件；明确仅影响后续对话

## 风险与待确认项

| 项 | 说明 |
|----|------|
| 对话与文件不一致 | 降级后模型上下文变短但文件仍为「未来态」；需在弹窗与成功 Toast 中明示 |
| 错误分类 | 哪些错误可降级 vs 必须硬失败，由 SPEC 枚举；本 PRD 仅要求「可判定 restore 失败」类支持降级 |
| Desktop / Mobile 一致性 | 双端弹窗文案与选项对齐 |
| 与隐藏/ Fork 关系 | 截断后 hidden 状态、Fork 会话不在本 PRD 范围 |
