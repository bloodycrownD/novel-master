# Mobile 会话工作区与 Worktree 性能 PRD

> **范围**：以 **Mobile 会话工作区** 体验为主；底层能力可在 core 复用，**CLI / Desktop 不纳入本迭代强制验收**。  
> **边界**：不含技术方案、接口、表结构、任务拆分（见后续 SPEC）。  
> **关联**：[event-bus-compaction-conditions](../event-bus-compaction-conditions/prd.md)（`SessionMacroCache` / `refresh-macros`）、[vfs-directory-nodes](../vfs-directory-nodes/prd.md)、[mobile-chat-thinking-and-vfs-sort](../mobile-chat-thinking-and-vfs-sort/prd.md)。

## 背景

用户在 Mobile「会话工作区」中管理较多文本文件（例如约 30 个、合计约 500KB）时，进入工作区、切换目录或刷新列表会出现可感知卡顿。

现状问题（产品层面）：

1. **工作区列表与 Agent 宏各走一套数据路径**：会话宏虽有缓存，但工作区文件列表不共用，重复扫描会话 VFS。
2. **列表加载读了文件正文**：文件管理器仅需路径、类型、规则与排序元数据，却与拼宏类似地加载全部 `content`，体感上「只是看个目录却很慢」。
3. **宏刷新成本偏高**：为生成 `{{.worktree}}` / `{{.filetree}}` 往往全量读正文，而大量文件在规则下可能为隐藏或仅展示文件名，无需正文。

本需求旨在让用户在典型创作会话规模下，**工作区浏览流畅**，且 **宏与工作区数据一致、不重复劳动**；不以「压缩存储正文」作为本迭代手段。

## 目标（含成功指标）

| 目标 | 成功指标 |
|------|----------|
| 工作区列表流畅 | 在 **约 30 个文本文件、合计约 500KB** 的会话中，进入「会话工作区」、切换当前目录、保存目录/文件规则后刷新列表，**无明显卡顿或长时间白屏**（以 Android 真机/模拟器主观流畅为准，对照迭代前录屏） |
| 列表不依赖正文 | 工作区列表展示与排序 **不读取** 文件 `content`；仅依赖路径、目录结构、mtime、worktree 规则等元数据 |
| 宏与工作区一致 | 同一会话、同一次 VFS/规则变更后，Agent 使用的宏文本与工作区所见规则/结构 **不出现长期不一致**（在约定刷新时机内对齐，见验收） |
| 单一缓存语义 | 不再存在「宏缓存一套、工作区再全量扫一遍」的 **并行重复刷新**；用户侧理解为「会话 worktree 快照」统一维护 |
| 宏按需使用正文 | 生成 worktree 宏正文时，**仅对需要展示正文的文件** 读取内容（如全文、front matter）；隐藏或仅文件名的项不读正文 |

## 用户与场景

| 用户 | 场景 |
|------|------|
| Mobile 写作用户 | 在会话中维护多章节/设定 txt、md，频繁打开「会话工作区」浏览、改规则、点进编辑 |
| Mobile 对话用户 | Agent 运行前后查看工作区文件是否与对话一致；压缩或 `refresh-macros` 后宏与列表仍可信 |
| 产品/验收 | 用「30 文件 / ~500KB」fixture 会话对比迭代前后进入工作区的流畅度 |

## 范围

### 包含范围

- Android Mobile **会话 scope** 工作区：文件列表加载、目录切换、规则保存后的列表刷新。
- **统一会话 worktree 快照**（产品概念）：一次物化结果同时支撑工作区列表与宏字符串；与现有 `refresh-macros`、Agent 开局刷新等时机对齐。
- **元数据路径**：工作区 `buildListRows` 等等价能力仅基于 path / mtime / 规则 / 目录结构，不读 `content`。
- **宏正文按需读**：`{{.worktree}}` 生成时按 display 状态决定是否读 `content`；`{{.filetree}}` 仅需树结构元数据。
- **失效策略（产品级）**：会话 VFS 增删改移、zip 导入、目录/文件规则变更、模板拉取等操作后，快照应在用户再次进入工作区或触发宏刷新前 **更新或作废**，避免列表与宏长期陈旧。
- Core 层能力若为实现上述体验所必需，可一并交付，但 **验收以 Mobile 会话工作区为准**。

### 不包含范围

- **将 VFS 正文改为 zip 等压缩存储**（本迭代不做；不以此作为性能验收项）。
- 大文件 **external 存储**、分片编辑、编辑器虚拟滚动（可另立需求）。
- **项目 / 全局** VFS 工作区的专项性能验收（若实现可顺带受益，不单独列达标）。
- CLI / Desktop 的专项性能指标与录屏对比。
- 修改 worktree 规则语义（head/tail/fill、排序字段含义不变）。
- 聊天消息列表分页/滚动（另属聊天迭代）。

## 核心需求

1. **工作区列表元数据化**：用户打开会话工作区或切换目录时，列表展示（名称、文件夹/文件、规则徽章、排序）仅基于元数据加载，**不得为列表扫描全部文件正文**。
2. **统一会话快照**：同一会话在同一「快照版本」下，工作区列表顺序/规则展示与宏 `filetree` / `worktree` 使用 **同一套物化结果**；避免宏已刷新而工作区仍全量重扫、或两套结果互相矛盾。
3. **宏刷新可感知且不过量**：`refresh-macros`、Agent 回合开始等既有刷新时机仍更新宏；刷新时 **不重复** 已为工作区做过的全量正文扫描；对隐藏/仅文件名的文件 **不读正文**。
4. **变更后快照失效**：用户在 Mobile 对工作区执行写文件、删改路径、改目录/文件规则、从模板拉取等操作后，再次查看工作区或依赖宏时，应得到 **更新后的列表与宏**，不得长期显示变更前状态。
5. **典型规模体验达标**：在约 30 个文本文件、合计约 500KB 的会话 fixture 上，迭代后进入工作区、切换目录、保存规则刷新的流畅度 **优于** 迭代前（团队内部 Before/After 录屏或共识评审）。
6. **与 core 排序/展示语义一致**：工作区当前目录子项顺序、规则灯、inclusion/display 文案仍与 `@novel-master/core` worktree 逻辑一致（不因性能优化改变用户可见语义）。

## 验收标准

### 工作区列表（Android · 会话 scope）

- **Given** 会话工作区存在约 30 个文本文件、合计约 500KB，且用户位于会话对话页  
  **When** 切换到「会话工作区」并等待列表首次展示完成  
  **Then** 列表在合理时间内出现（无长时间仅「加载中…」）；滚动与点击目录 **无明显掉帧卡顿**（对照迭代前同一 fixture）。

- **Given** 同上 fixture，用户位于工作区某一目录  
  **When** 进入子目录再返回上级、或修改目录排序规则并保存  
  **Then** 列表顺序/规则展示更新；整个过程 **不出现** 与迭代前相比更长的阻塞感。

- **Given** 同上 fixture  
  **When** 仅打开工作区浏览、不打开任一文件编辑器  
  **Then** 产品/开发可通过约定观测方式确认：**列表加载路径未读取文件正文**（验收依赖 SPEC 约定之观测点或测试替身，PRD 要求行为成立）。

### 统一快照与宏（Android · 会话 scope）

- **Given** 用户刚在 Agent 回合开始或 `refresh-macros` 之后查看工作区  
  **When** 打开会话工作区根目录列表  
  **Then** 目录结构、规则状态与当前宏 `filetree` 所反映的会话树 **一致**（无「宏已新、列表仍旧」超过一次手动刷新仍不更新的情况）。

- **Given** 目录规则使多数文件为隐藏或仅展示文件名  
  **When** 触发宏刷新  
  **Then** 刷新耗时不应随「隐藏/仅名」文件数量线性读全文而恶化；**仅**对需要正文展示模式的文件读取 `content`（验收依赖 SPEC 观测或测试）。

- **Given** 用户在工作区新建或修改一个文件并保存  
  **When** 返回工作区列表，或发起下一轮 Agent（会刷新宏）  
  **Then** 列表能看到新文件或变更；宏中的 worktree/filetree **包含** 该变更（在对应刷新时机之后）。

### 明确不验收

- **Given** 本迭代范围  
  **When** 评估存储层是否采用 zip 压缩正文  
  **Then** **不在** 本 PRD 验收范围内。

- **Given** CLI 或 Desktop 客户端  
  **When** 仅使用非 Mobile 入口操作同一会话  
  **Then** **不强制** 本 PRD 的性能与快照验收（除非另行立项）。

## 风险与待确认项

| 项 | 说明 |
|----|------|
| 「无明显卡顿」 | 主观标准；建议迭代内固定 fixture 设备与 Before/After 录屏作为评审附件 |
| 项目/全局工作区 | 首期不验收；若用户高频使用项目 VFS，可后续单列 PRD |
| 快照与并发 | Agent 工具写文件与用户同时编辑时的最终一致性，在 SPEC 中定义边界（PRD 要求最终一致、无长期陈旧） |
