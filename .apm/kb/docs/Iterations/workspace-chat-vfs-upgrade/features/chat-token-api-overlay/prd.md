# chat-token-api-overlay Feature PRD

## 背景

父级要求 Token **一套计算引擎**。现状：

- Mobile 对话 meta 顶栏、Desktop 工作区页脚、压缩触发均走**本地** `countPromptLlmInput`，按模型「计数方式」（auto / 指定族 / heuristic）选择解析器。  
- OpenAI / Anthropic / Gemini 等已解析 API `usage` 并进入 Agent 回合摘要，但**聊天 UI 与压缩均不消费**。  
- 模型配置**没有**、也**不应新增**独立的「API 计数」选项。  
- Desktop 页脚在发送后常不随消息刷新，即使用上 API 数也可能看不见。

历史 PRD（`model-aware-token-counting`）将 UI/压缩统一为本地 tokenizer；本 feature **增量**：在同一产品口径上，**有可用 API prompt 计数则采用，否则本地**——展示与压缩必须读同一结果，禁止双轨。

## 目标（含成功指标）

| 目标 | 成功指标 |
|------|----------|
| 单引擎 | 任意时刻，聊天占用展示与压缩判断使用的「当前占用」来自**同一决策结果**（同会话、同 Agent 上下文） |
| 优先 API | 最近一次成功回合若提供可用的 prompt 侧 token 数，展示与压缩均采用该结果（在有效期内） |
| 回退本地 | usage 缺失 / 不可用 → 展示与压缩均回退本地计数方式；**不**为此增加配置项 |
| 双端可见 | Mobile 顶栏与 Desktop 页脚在回合成功后都能看到更新后的占用（含 Desktop 发送后刷新） |
| 配置面不变 | 「计数方式」选项表仍无独立 `api` 项；auto 仍表示按模型名匹配**本地**族（文案可不误导为「自动选 API」） |

## 用户与场景

| 用户 | 场景 |
|------|------|
| 对话用户 | 发完一轮后顶栏/页脚数字与「是否触发压缩」一致，且尽量贴近服务商真实 prompt 计数 |
| 中转站用户 | 上游不返回 usage → 静默用本地估算，压缩与展示仍一致 |
| 改草稿用户 | 改消息/换模型/压缩后，占用按约定失效并重新由引擎计算（见核心需求） |

## 范围

### 包含范围

1. 统一「当前占用」决策：有可用 API prompt tokens → 用；否则 → 本地（尊重模型「计数方式」）。  
2. **聊天展示**（Mobile 顶栏、Desktop 页脚）与 **压缩触发** 均消费上述结果。  
3. Desktop：**发送/回合结束后页脚必须刷新**，使用户能看到更新。  
4. 无 usage、字段不可用时的回退与可观测；**采用 API 占用时两端统一为非估算**（展示种类为 `api`），本地回退仍带原 counter 种类与估算标记。  
5. 不新增 `tokenCounterMode: api`。

### 不包含范围

- 废除本地 tokenizer / NMTP 驱动（发送前与无 usage 时仍需要）  
- 把「auto」产品文案改成「自动在 API 与解析器间选择」若易与「按模型名匹配」冲突——以「单引擎优先 API」行为为准，文案 Spec 阶段微调即可  
- CLI `prompt render --tokens` 强制改读 API（可继续本地；若读统一引擎须与聊天同口径，不强制本 feature）  
- 将 usage 作为独立业务字段长期展示「completion tokens」明细（非本 feature 目标）  
- iOS 强制验收

## 核心需求

1. **一套结果，两处消费**  
   展示与压缩不得长期一个用 API、一个用本地。

2. **可用性判定**  
   以「存在可用的 prompt 侧计数」为准（典型为 API `promptTokens` 有值）。  
   **不得**仅用「全为 0 = 不支持」作为唯一信号（0 可能是合法空上下文；缺失多为 `undefined`）。

3. **生命周期**  
   - 发送前 / 尚无成功 usage：用本地。  
   - **仅**回合正常完成（非取消 / 非触顶步数等）且 usage 可用：引擎切换到该 API 结果，展示与压缩一致；否则丢弃该会话上的 API 占用缓存并回退本地。  
   - **含失败**：非取消类异常导致回合失败时，同样须丢弃该会话 API 占用缓存并回退本地（现网失败路径不到达「正常结束」事件，细则见 SPEC）。  
   - 用户改动会改变 prompt 的操作（如改/删消息、换模型、手动压缩成功等）：须重新计算；有新 usage 再用新 usage，否则本地。  
   - 技术侧：压缩评估上下文须携带会话 id，以便与展示共用按会话分桶的占用结果（细则见 SPEC）。  

4. **多步 tool 回合**  
   产品默认采用**该次用户可见回合内、可用于代表「当前 prompt 占用」的约定值**（例如末轮带 prompt 的 usage，或文档化的累加规则）；展示与压缩必须用**同一约定**，细则在 SPEC 写死并验收。

5. **Desktop 刷新**  
   回合结束后页脚占用数字更新，纳入本 feature 验收（不得「数已更新但 UI 不刷新」）。

## 验收标准

- [ ] Given 模型返回可用 `promptTokens`，When 回合成功，Then Mobile 顶栏与 Desktop 页脚占用与压缩是否触发**基于同一占用值**  
- [ ] Given 上游不返回 usage，When 回合成功，Then 展示与压缩均回退本地，且二者仍一致；设置中**无**新增「API 计数」项  
- [ ] Given Desktop 会话，When 发送成功，Then 页脚 token 区域在无需切换会话的情况下更新  
- [ ] Given 用户删除一条会改变 prompt 的消息，When 占用刷新，Then 不再错误沿用过期的 API 值（或按约定立即重算）  

## 风险与待确认项

| 项 | 说明 |
|----|------|
| 多轮 tool 的 usage 合并 | 末轮 vs 累加 → SPEC 必须选定一种并测 |
| Anthropic 流式 usage 字段不完整 | 可能导致「本应可用却回退本地」；验收允许回退，但双端须一致 |
| 百分比分母 | 仍为模型上下文上限；本 feature 不改分母语义 |
